(목표: “데이터 꼬임이 와도 잘못된 데이터는 무시하고, 자동으로 정상화(복구)된다”)

작업지시서: TCP ASCII Robot 통신 “소프트 꼬임(프레임/윈도우 불일치)” 자동 정상화
0) 목표/배경

현재 증상: 소켓은 연결되어 있어도(Connected 유지) 20B/20C RX invalid, window incomplete, checksum mismatch, chunk anomaly가 연속 발생하면 폴링이 망가지고 UI가 “끊김/Failed”로 보이며 원복(정상화)이 안 되는 구간이 발생.

원인 가설(로그 근거): 요청/응답 스트림이 뒤섞이거나(동시 RoundTrip / 잔여버퍼) 소프트 리셋 후에도 프레임 경계 재동기화(resync)가 안 되어 쓰레기 응답을 계속 읽는 상태.

1) 작업 범위(파일 단위)
1순위: RoundTrip 계층 Resync + 직렬화(동시성 차단)

파일: Services/Robot/RobotService.Core.cs (특히 RoundTripAsync)

목적:

동시에 여러 명령이 날아가면서 응답이 섞이지 않게 RoundTrip을 1개씩만 수행

프레임/체크섬/윈도우가 깨졌을 때 버퍼를 비우고 정상 프레임 경계로 재동기화

2순위: soft-stuck 연속 시 “강제 스트림 리셋(Disconnect→Connect)” 정책 고도화

파일(책임 위치에 따라 택1 또는 둘다):

Services/Runtime/PollingService.cs (루프에서 stuck 감지/요청)

Services/Runtime/SystemController.cs (리셋 실행 책임)

목적:

Invalid/WindowIncomplete/Checksum/Timeout 등 “소프트 꼬임”을 연속 카운트로 감지

일정 조건이면 Poll 루프를 멈춘 상태에서 Disconnect→Connect 수행

리셋 후 resync/flush를 반드시 수행하고 루프 재개

보너스(현장 체감 개선): MovePrecheck DI 읽기 1~2회 재시도

파일: Services/Runtime/MovePrecheck.cs

목적: IO 갱신 직전에 한 번 꼬였을 때 즉시 Fault로 가지 않고 1~2회만 재시도.

2) 구현 요구사항(핵심 설계 원칙)

RoundTrip은 반드시 직렬화

SemaphoreSlim(1,1) 또는 전역 lock으로 RoundTripAsync 단일 진입 보장.

폴링(IO/Axis/System) + UI 명령(WriteOutput 등)이 동시에 호출되어도 응답이 섞이지 않게.

소프트 리셋은 “완전 정지 → 리셋 → 재동기화 → 재개” 순서

잘못된 예: 폴링이 계속 돌고 있는 상태에서 Disconnect/Connect를 여러 스레드가 동시에 수행

올바른 예:

Poll 루프들(최소 IO)을 멈춤(취소 토큰/플래그)

Disconnect

Connect

Flush/Resync

Poll 재시작

UI 메시지 분리(진짜 소켓끊김 vs 통신불량)

기존 OnConnectionLost() 메시지는 유지하되,

Robot.Connected == false → “소켓 연결 끊김”

Robot.Connected == true → “통신 프레임 수신 실패(연속)”

(이건 기능이라기보다 운영/현장 커뮤니케이션 품질 개선)

3) 상세 작업 항목
[Task-1] RobotService.RoundTripAsync 직렬화 + Resync(필수)

수정 내용

private readonly SemaphoreSlim _rtLock = new(1,1); 추가

RoundTripAsync 시작에서 _rtLock.WaitAsync(ct) 후 finally에서 Release

Resync 정책(권장 기본값)

N = 30 (최대 30라인까지 “정상 프레임 후보”를 스캔)

T = 300ms (resync 전체 제한 시간)

동작:

Checksum mismatch, Malformed frame, 20B/20C RX invalid, window incomplete 등 발생 시:

즉시 예외로 끝내지 말고

ReadLineAsync로 라인을 추가로 읽으며 “정상 프레임 헤더(#/! + 길이/ID 규칙)”에 맞는 라인을 찾음

정상 후보가 나오면 그 라인을 기준으로 복구 시도(또는 “다음 요청부터 정상”을 기대하고 내부 상태 리셋)

제한을 초과하면 “soft-stuck 카운트 증가”로 넘기고 상위에서 강제 리셋 트리거 가능하게 함

포인트: resync는 “무한 루프”가 되면 안 되고, 반드시 N/T로 제한해야 합니다.

추가 로깅(필수)

roundtrip_enter/exit, roundtrip_resync_enter/exit, roundtrip_resync_found, roundtrip_resync_fail

필드: cmd, reqId, exMsg, linesScanned, elapsedMs

[Task-2] soft-stuck 연속 감지 + 강제 리셋(Disconnect→Connect) (필수)

감지 대상(soft error 분류)

InvalidOperationException:

“20B RX invalid”

“20C RX invalid”

“window incomplete …”

“Checksum mismatch”

“Malformed frame”

Timeout 패턴(있는 경우):

OperationCanceledException / TimeoutException 계열을 별도 카운트 (정책 결정)

리셋 트리거 조건(권장 기본값)

K = 12 (soft error 연속 12회면 리셋 요청)

지금 로그는 basePeriod 100ms / backoff 800ms까지 가며 연속이 쉽게 쌓임 → 12면 충분히 빠르게 잡힙니다.

Cooldown:

1초 가능. 다만 “연속 리셋 폭주” 방지 장치가 필요해서 계단식 권장

첫 리셋: 1초

2회 연속 리셋: 3초

3회 이상: 5초

(너가 원한 “1초”를 살리면서도 폭주를 막는 안전장치)

리셋 실행 규칙(필수)

리셋 수행 중 재진입 방지: Interlocked.CompareExchange로 “ResetInProgress” 가드

리셋 단계 로그:

robot_soft_reset_enter(reason, consec, cooldownMs)

robot_soft_reset_stop_polling

robot_soft_reset_disconnect

robot_soft_reset_connect

robot_soft_reset_resync_flush

robot_soft_reset_exit(success/fail)

Flush/Resync 위치(필수)

Disconnect/Connect 후에 반드시 “읽기 버퍼 정리(가능하면)” + 짧은 resync 수행

현실적으로 NetworkStream의 “flush”는 애매하니,

Connect 직후 100~200ms 동안 ReadAvailable(있으면 읽어버림)

또는 ReadLineAsync를 짧은 타임아웃으로 몇 번 호출해 버퍼를 비움

(이 단계가 없으면 “재연결했는데도 쓰레기 라인부터 읽는” 문제가 반복됨)

[Task-3] MovePrecheck DI8 등 Read 1~2회 재시도 (선택/권장)

DI 읽기 실패 시 즉시 Fault 대신:

100ms 간격 2회 재시도

그래도 실패면 기존 로직대로 Fault

목적: soft 꼬임이 “딱 1~2회” 튀는 경우 현장 체감 크게 좋아짐

4) 테스트/검증 시나리오(시뮬레이터 기반)

bad_checksum

truncated line

duplicate/burst (응답 2개 붙여 보내기)

성공 기준(완료 조건)

soft 오류가 연속으로 발생해도 시스템이 5초 이내 정상 폴링으로 복구해야 함

io_poll_recovered가 반드시 다시 나와야 함

UI에서 “끊김/Failed”가 떠도 자동으로 Connected로 돌아올 수 있어야 함
(또는 최소 “통신불량 복구중” 같은 중간 상태 표시)

리셋 폭주 금지

30초 동안 soft 꼬임 지속 주입 시

soft reset이 무한 반복하지 않고, 쿨다운(backoff)이 적용되어야 함

RoundTrip 동시 호출(폴링 + WriteOutput + Reset) 상황에서도

응답이 섞여 RX invalid가 폭발하지 않아야 함(직렬화로 억제)
